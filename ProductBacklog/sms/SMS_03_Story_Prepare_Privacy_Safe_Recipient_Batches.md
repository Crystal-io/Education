# Story: Prepare Privacy-Safe Recipient Batches

## 1. User Story

As the **CRM System**,  
I want to prepare SMS campaign recipient batches using privacy-safe recipient references generated by the Player Data Service,  
So that eligible recipients can be processed through the campaign pipeline without exposing phone numbers or other personal data across CRM modules.

---

## 2. Business Context

This Story supports the SMS Campaign Delivery Pipeline Epic by preparing recipients for downstream short link allocation and SMS sending.

The key business and system problem is that large SMS campaigns require recipient-level processing, but personal data such as phone number, name, email, country-specific identifiers or player profile details should not be passed through every CRM module, message broker, integration service or log stream.

### Selected Compliance-Friendly Approach

The recommended approach is **data minimization through privacy-safe recipient references**.

Instead of passing personal data through the full pipeline, the Player Data Service generates a **scoped, non-reversible recipient reference** for each eligible player.

This reference may be implemented as an opaque token or HMAC-based user hash, but it must be:

- generated by the Player Data Service;
- scoped to campaign, channel and tenant;
- time-bound or lifecycle-bound;
- non-reversible outside the trusted Player Data / Sender boundary;
- resolvable only by authorized services at the final sending step.

This is preferred over passing encrypted personal data through the whole pipeline. Encryption protects data in transit and at rest, but still spreads personal data and key-management responsibility across multiple services. A recipient reference keeps the pipeline lightweight and reduces compliance exposure.

## 3. User / Role

- **Primary role:** CRM System / Campaign Processing Service
- **Secondary roles:** CRM Manager, Marketing & Retention Manager, Operations User
- **System actors:** Player Data Service, Campaign Processing Service, Message Broker / Queue, SlimLink Integration Service, SMS Sender / SMS Router
- **Tenant / market limitations:** Recipient eligibility and consent rules may depend on tenant, market, product, communication channel and regional compliance rules.

## 4. Preconditions

- SMS campaign exists in CRM and has been created with selected audience, message content and campaign long URL.
- Campaign is in a state that allows recipient preparation.
- Selected audience or segment is available and versioned.
- Player Data Service owns player personal data, phone number availability and consent information.
- SMS consent rules are configured for the relevant tenant / market / product.
- Campaign Processing Service is allowed to request privacy-safe recipient references from Player Data Service.
- Batch size and downstream processing limits are configured.

---

## 5. Postconditions

- Eligible campaign recipients are prepared as batches using privacy-safe recipient references.
- No phone numbers, names, emails or other direct personal data are stored in campaign processing batches.
- Each recipient reference is mapped to campaignId and batchId in CRM campaign processing context.
- Excluded recipients are counted by exclusion reason without exposing unnecessary personal data.
- Recipient batches are ready for short link allocation and SMS preparation.
- Campaign status is updated to reflect recipient preparation result.
- Logs, audit events and metrics are created for recipient batch preparation.

---

## 6. Main Flow

1. CRM Manager launches or schedules an SMS campaign.
2. Campaign Processing Service receives campaignId, selected audience reference and campaign configuration.
3. Campaign Processing Service requests eligible SMS recipients from Player Data Service.
4. Player Data Service evaluates recipient eligibility based on:
   - audience membership;
   - player status;
   - phone number availability;
   - SMS consent;
   - tenant / market / regional rules;
   - suppression / blacklist / communication restrictions.
5. Player Data Service generates a privacy-safe recipient reference for each eligible player.
6. Player Data Service returns eligible recipient references and allowed non-PII processing metadata to Campaign Processing Service.
7. Campaign Processing Service splits eligible recipient references into configured batches.
8. Campaign Processing Service creates Campaign Batch records with initial status `Prepared`.
9. Campaign Processing Service publishes recipient batch processing event(s) for downstream short link allocation.
10. Campaign status is updated to `Processing` or `Recipients Prepared`, depending on the existing campaign state model.
11. Processing metrics and audit events are recorded.

---

## 7. Alternative Flows

### A1. Player Data Service is unavailable

1. Campaign Processing Service fails to receive recipient references.
2. System does not create incomplete recipient batches.
3. Campaign status is set to `Processing Failed` or remains `Processing` with retry scheduled, depending on retry policy.
4. Operations alert is triggered if configured threshold is reached.
5. Error is logged with campaignId and correlationId.

### A2. No eligible recipients found

1. Player Data Service returns zero eligible recipient references.
2. System does not create downstream batches.
3. Campaign status is set to `No Eligible Recipients` or `Failed Validation` depending on agreed status model.
4. CRM Manager can see a user-friendly reason summary.

### A3. Some recipients are excluded

1. Player Data Service returns eligible recipient references and aggregated exclusion counters.
2. System continues processing eligible recipients.
3. Excluded recipients are not sent to SlimLink or SMS Sender.
4. CRM Manager / Operations can see exclusion summary where permitted.

### A4. Recipient reference generation fails for part of the audience

1. Player Data Service returns successful references and failed reference-generation records or aggregated failure counts.
2. Campaign Processing Service creates batches only for successful references.
3. Failed records are stored as recipient preparation failures where permitted.
4. Campaign may continue as `Partially Prepared` / `Partially Failed` depending on threshold rules.

### A5. Recipient reference expires before final SMS sending

1. SMS Sender cannot resolve the recipient reference at final sending step.
2. SMS is not sent for this recipient.
3. Recipient status is set to `Failed` with reason `Recipient Reference Expired`.
4. Operations can reprocess the recipient only through an authorized recovery flow.

## 8. Edge Cases

| Case | Expected behavior |
|---|---|
| Audience contains duplicate players | Player Data Service or Campaign Processing Service deduplicates recipients before batch creation. |
| Player has no valid phone number | Recipient is excluded from SMS processing. |
| Player has no SMS consent | Recipient is excluded from SMS processing. |
| Player is blocked / self-excluded / blacklisted | Recipient is excluded according to business and compliance rules. |
| Same player appears in multiple audience sources | Recipient is included only once per campaign. |
| Recipient reference is malformed | Record is rejected and not included in downstream batch. |
| Batch size exceeds configured limit | System splits recipients into multiple batches. |
| Campaign is cancelled during recipient preparation | Batch creation stops; already created batches are marked as cancelled if not yet processed. |
| Player data changes after batch preparation | Downstream sender should resolve final phone/contact state at the last responsible moment; invalid recipients are skipped. |
| Token / hash generation key is rotated | New references use the active key; existing references remain resolvable until TTL or campaign completion. |

## 11. Acceptance Criteria

### AC-001: Eligible recipients are prepared without exposing PII

Given an SMS campaign with selected audience,  
When recipient preparation is started,  
Then the system requests privacy-safe recipient references from Player Data Service,  
And campaign processing batches do not contain phone number, name, email or other direct personal data.

### AC-002: Only SMS-eligible recipients are included

Given selected audience contains players with different consent and phone states,  
When Player Data Service prepares recipient references,  
Then only players with valid SMS consent and valid phone availability are returned as eligible recipients,  
And ineligible players are excluded from downstream processing.

### AC-003: Recipients are deduplicated per campaign

Given the same player appears more than once in the selected audience,  
When recipient batches are prepared,  
Then the player is included only once in the campaign recipient list.

### AC-004: Recipient batches are created within configured limits

Given the number of eligible recipients exceeds the configured batch size,  
When recipient preparation is completed,  
Then the system creates multiple batches,  
And no batch exceeds the configured size limit.

### AC-005: Downstream events contain only allowed data

Given recipient batches are created,  
When the system publishes recipient batch events,  
Then events contain campaignId, batchId, recipientRef and allowed non-PII metadata only,  
And events do not contain phone numbers or other direct personal data.

### AC-006: Recipient preparation retry does not duplicate batches

Given recipient preparation request is retried after a temporary failure,  
When the same campaign preparation request is processed again,  
Then the system does not create duplicate recipient batch records for the same campaign and audience version.

### AC-007: No eligible recipients are handled clearly

Given no players from the selected audience are eligible for SMS,  
When recipient preparation is completed,  
Then no downstream batches are created,  
And campaign status or validation result clearly indicates that there are no eligible recipients.

### AC-008: Personal data is resolved only at final sending boundary

Given an eligible recipient has a generated recipientRef,  
When the campaign moves through SlimLink allocation and batch processing,  
Then personal data remains hidden from intermediate services,  
And only authorized SMS Sender / Player Data components can resolve the recipientRef for final SMS delivery.

## 17. API / Event Details

### Endpoint / Service Operation

- **Endpoint / operation:** `PrepareSmsRecipientReferences` / TBD by Player Data Service contract
- **Method:** TBD
- **Request:** campaignId, tenantId, audienceRef, audienceVersion, channel = SMS, campaign scope, requestId
- **Response:** preparationId, eligible recipient references, aggregated exclusion counters, failed count, status
- **Error codes:** TBD by Player Data Service contract

### Event

- **Event name:** `sms.recipient_batch.prepared`
- **Event payload:** campaignId, tenantId, batchId, recipientRefs, recipientCount, processingScope, correlationId
- **Payload restriction:** No phone numbers, names, emails or direct personal data.

### Idempotency

- Idempotency key should be based on campaignId + audienceVersion + preparationRequestId.
- Retried preparation request must not create duplicate recipient batches.
- Reprocessing must be explicitly authorized and versioned.

### Retry behavior

- Retry only for temporary Player Data Service or infrastructure failures.
- Do not retry if campaign is cancelled, audience is invalid or user has no permission.
- Retry attempts must be logged and monitored.
