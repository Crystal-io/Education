# Task: Define Privacy-Safe Recipient Batch Contract

## 1. Context

This Task supports the Story **Prepare Privacy-Safe Recipient Batches** under the Epic **SMS Campaign Delivery Pipeline with SlimLink Short Links**.

The SMS campaign pipeline must process large recipient lists across multiple CRM services without exposing unnecessary personal data. The existing campaign flow should not pass phone numbers, names, emails or other PII through every intermediate module.

Instead, the pipeline should use a **privacy-safe recipient reference** prepared by the service responsible for player / user data. This reference may be called `recipientRef`, `userHash`, or similar internally, but from a requirements perspective it must be treated as an opaque technical identifier.

The final SMS Sender should resolve the actual phone number only at the last responsible step, within a controlled service boundary and according to SMS consent and access rules.

---

## 2. Objective

Define the recipient batch contract that allows CRM campaign processing to work with large SMS audiences in a privacy-safe and compliance-friendly way.

As a result:

- Campaign Processing receives batches containing recipient references, not raw PII.
- Intermediate services process only technical identifiers and campaign metadata.
- Player Data Service remains the source of truth for player identity, phone number and consent data.
- SMS Sender resolves personal data only when it is required for actual SMS delivery.
- Logging, monitoring and retry flows do not expose player phone numbers or other PII.

## 7. Implementation Notes

### Recommended approach

Use an **opaque recipient reference** generated by the Player Data Service rather than passing raw or encrypted personal data through the CRM pipeline.

This is the preferred approach because:

- intermediate services do not need phone numbers to perform campaign processing;
- PII exposure is minimized by design;
- logs, retries, queues and operational dashboards remain safer;
- access to actual phone numbers is limited to the final sender boundary;
- compliance risk and operational blast radius are reduced;
- system load is reduced because intermediate services process lightweight references instead of full player data.

### Why not raw PII

Passing phone numbers, names or emails through multiple modules increases:

- GDPR / compliance exposure;
- access-control complexity;
- logging and masking risks;
- data leakage risk in queues and troubleshooting tools;
- regression risk when new services are added to the pipeline.

### Why not encrypted PII as the main design

Encrypted PII is better than raw PII, but it is still less preferred for this pipeline because:

- encrypted payloads are still personal data from a compliance perspective;
- key management and rotation become part of every pipeline component;
- every downstream service must understand encrypted payload rules;
- troubleshooting becomes more complex;
- accidental logging of encrypted data may still be treated as sensitive;
- services that do not need PII still carry it unnecessarily.

Encrypted PII may be acceptable only for specific point-to-point transfers where the recipient service genuinely needs the data and key management is controlled.

### Preferred reference characteristics

The recipient reference should be:

- opaque for all intermediate services;
- non-reversible outside the Player Data / Sender boundary;
- scoped to tenant and campaign or another agreed security scope;
- stable enough for campaign processing, retries and reconciliation;
- not derived directly from phone number using a simple unsalted hash;
- not usable to infer player identity without authorized lookup;
- logged only as a technical identifier.

- **Request / response:** Logical contract should support campaignId, batchId, recipientRef / userHash, eligibility status and non-PII metadata

- **Retry:** Failed batch preparation or reference resolution may be retried if the error is temporary

## 9. Data / DB Details

### Allowed Data in Intermediate Batch

| Data Element | Notes |
|---|---|
| `campaignId` | Required for traceability. |
| `batchId` | Required for batch processing and retries. |
| `recipientRef` / `userHash` | Opaque recipient reference generated by Player Data Service. |
| `tenantId` / `brandId` | Allowed if needed for routing and permissions. |
| `market` / `region` / `language` | Allowed only if needed for campaign processing and not identifying by itself. |
| `eligibilityStatus` | Allowed as non-PII campaign-processing status. |
| `exclusionReasonCode` | Allowed as standardized non-PII reason code. |
| Processing status | Allowed: Pending, Prepared, Failed, Sent, etc. |

### Prohibited Data in Intermediate Batch

| Data Element | Reason |
|---|---|
| Phone number | Only final SMS Sender should access delivery phone number. |
| Full name | Not required for pipeline processing. |
| Email address | Not required for SMS pipeline processing. |
| Date of birth | Not required for campaign processing. |
| Address / location precision | Not required unless explicitly approved and minimized. |
| Raw player profile payload | Violates data minimization principle. |
| Consent evidence details | Store / resolve in source system; pipeline may receive only eligibility result. |

### Data Storage Notes

- Campaign Processing may store recipientRef and processing status.
- Mapping between recipientRef and actual player identity should stay in Player Data Service or another approved secure boundary.
- SMS Sender may temporarily resolve recipientRef into phone number only for sending.
- Any temporary resolved PII must follow retention, masking and audit rules.

## 13. Completion Criteria / Acceptance Criteria

### AC-001: Recipient batch contract excludes PII

Given the recipient batch contract is reviewed,  
When intermediate batch payload is defined,  
Then it contains recipientRef / userHash and campaign processing metadata only,  
And it does not contain phone number, name, email or raw player profile payload.

### AC-002: Player Data Service owns recipient reference generation

Given a campaign audience is prepared,  
When eligible recipients are requested,  
Then Player Data Service provides privacy-safe recipient references,  
And Campaign Processing does not generate references from raw PII independently.

### AC-003: SMS Sender resolves recipient reference only at final step

Given SMS messages are ready for sending,  
When SMS Sender processes a recipientRef,  
Then it resolves the actual phone number only within the authorized sender boundary,  
And unresolved recipients are marked as failed without exposing PII.

### AC-004: Logging is PII-safe

Given batch processing, retry or failure occurs,  
When logs and monitoring events are produced,  
Then they include campaignId, batchId, recipientRef and safe reason codes,  
And they do not include phone number or other PII.

### AC-005: Retry and reconciliation use recipientRef

Given batch processing is retried,  
When the same recipient is processed again,  
Then the system uses campaignId + batchId + recipientRef for retry / reconciliation,  
And does not require raw PII for duplicate prevention.

### AC-006: Invalid recipients are excluded safely

Given a recipient does not meet SMS eligibility, phone availability or consent requirements,  
When the batch is prepared,  
Then the recipient is excluded or marked failed with a non-PII reason code.

## 15. Risks

| Risk | Impact | Mitigation |
|---|---|---|
| Raw PII is added to batch payload for convenience | Compliance and leakage risk | Contract explicitly prohibits PII in intermediate batches. |
| Simple hash is derived from phone number | Re-identification risk | Use opaque reference / HMAC with controlled secret and scope, not plain hash. |
| Too many services can resolve recipientRef | Access-control risk | Limit resolution to SMS Sender and approved Player Data boundary. |
| Reference is not stable enough for retries | Retry / reconciliation issues | Define campaign-scoped stability and idempotency rules. |
| RecipientRef becomes globally trackable across campaigns | Privacy risk | Use scoped reference strategy where required. |
| Sender cannot resolve reference at send time | SMS delivery failures | Add resolution failure handling and pre-release integration test. |
| Logs accidentally include resolved phone numbers | Data leakage risk | Add logging requirements, masking and QA checks. |
