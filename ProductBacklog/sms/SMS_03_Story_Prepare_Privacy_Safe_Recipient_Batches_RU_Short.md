# Story: Подготовка privacy-safe recipient batches для SMS-кампании

## Краткое назначение

Сокращенная версия Story для интервью.  
Фокус: показать, как BSA описывает подготовку batch для SMS-кампании без протаскивания персональных данных через весь CRM pipeline.

---

## 1. User Story

**As the CRM System,**  
I want to prepare SMS campaign recipient batches using privacy-safe recipient references generated by the Player Data Service,  
**So that** eligible recipients can be processed through the campaign pipeline without exposing phone numbers or other personal data across CRM modules.

По-русски:

**Как CRM System,**  
я хочу подготовить batch-и получателей SMS-кампании, используя безопасные recipient references / user hashes, сформированные сервисом пользовательских данных,  
**чтобы** обрабатывать получателей в campaign pipeline без передачи phone number, email, имени и других персональных данных между CRM-модулями.

---

## 2. Business Context

Эта Story поддерживает Epic **SMS Campaign Delivery Pipeline with SlimLink Short Links**.

Основная проблема: SMS-кампания требует обработки большого списка игроков, но персональные данные не должны проходить через все сервисы pipeline: CRM Campaign, queue/broker, SlimLink integration, monitoring и logs.

Ценность решения:

- снижает compliance-риск и соблюдает data minimization;
- уменьшает нагрузку на downstream-сервисы;
- снижает риск утечки PII через logs/events/monitoring;
- оставляет персональные данные только в сервисах, которые реально должны с ними работать;
- упрощает аудит и контроль доступа.

---

## 3. Selected Approach

Выбранный подход: **privacy-safe recipient reference**.

Player Data Service, который владеет персональными данными игрока, формирует для каждого eligible recipient специальный `recipientRef` / `userHash`.

Этот reference:

- не содержит phone number, email, name или другие прямые PII;
- генерируется только Player Data Service;
- привязан к tenant / campaign / channel;
- не должен быть обратимо расшифрован промежуточными CRM-сервисами;
- резолвится в реальные contact details только на финальном шаге отправки SMS.

### Почему не передавать PII в зашифрованном виде

Передача encrypted PII технически возможна, но хуже как основной вариант:

- персональные данные всё равно проходят через многие модули;
- появляется управление ключами, rotation, access control, decrypt permissions;
- растет риск случайного попадания PII в logs/events/debug traces;
- каждый сервис в цепочке становится security-sensitive.

Поэтому лучший вариант для этого pipeline: **не передавать PII вообще**, а передавать только scoped recipient reference.

## 4. User / Role

| Role | Responsibility |
|---|---|
| CRM System / Campaign Processing Service | Инициирует подготовку batch и downstream processing. |
| Player Data Service | Проверяет eligibility, consent, phone availability и генерирует recipientRef. |
| SMS Sender | На финальном шаге резолвит recipientRef в реальные contact details. |
| CRM Manager | Запускает кампанию и видит агрегированный результат. |
| Operations User | Видит ошибки batch processing без доступа к лишним PII. |

---

## 5. Preconditions

- SMS campaign создана в CRM с выбранной аудиторией, текстом сообщения и long URL.
- Player Data Service содержит данные по игрокам, phone availability и SMS consent.
- Для рынка / tenant настроены consent и communication rules.
- Campaign Processing Service имеет право запросить recipient references.
- Настроены batch size и downstream limits.

---

## 6. Postconditions

- Eligible recipients подготовлены в batch-ах через `recipientRef` / `userHash`.
- Campaign pipeline не хранит phone number, email, name или другие direct PII.
- Каждый batch связан с `campaignId`, `batchId` и набором recipient references.
- Excluded recipients учтены агрегированно по причинам исключения.
- Batch-и готовы для short link allocation и дальнейшей SMS-отправки.

---

## 7. Main Flow

1. CRM Manager запускает SMS campaign.
2. Campaign Processing Service получает `campaignId`, audience reference и campaign configuration.
3. Campaign Processing Service запрашивает у Player Data Service список eligible recipients.
4. Player Data Service проверяет audience membership, phone availability, SMS consent, blacklist/suppression rules и regional restrictions.
5. Для каждого eligible player Player Data Service генерирует `recipientRef` / `userHash`.
6. Campaign Processing Service получает только recipient references и allowed non-PII metadata.
7. System делит recipients на batch-и согласно configured batch size.
8. Для каждого batch создается processing record со статусом `Prepared`.
9. Batch-и передаются дальше в pipeline для short link allocation / SMS preparation.
10. System логирует результат подготовки без раскрытия PII.

---

## 8. Alternative Flows / Edge Cases

| Case | Expected Behavior |
|---|---|
| Player Data Service unavailable | Batch-и не создаются; campaign получает failed/retry status; Operations получает alert. |
| No eligible recipients | Campaign не уходит в downstream processing; CRM Manager видит понятную причину. |
| Some recipients excluded | Eligible recipients продолжают обработку; исключенные считаются агрегированно. |
| Duplicate players in audience | Игрок включается в кампанию только один раз. |
| RecipientRef expired before sending | SMS не отправляется; recipient получает failed status; возможен controlled reprocessing. |
| Campaign cancelled during preparation | Создание batch останавливается; созданные batch-и переводятся в cancelled/skipped. |

## 9. Functional Requirements

| FR ID | Requirement |
|---|---|
| FR-001 | System shall request SMS-eligible recipient references from Player Data Service. |
| FR-002 | System shall not store phone numbers, names, emails or direct PII in campaign batches. |
| FR-003 | Player Data Service shall generate campaign/channel-scoped recipient references. |
| FR-004 | System shall include only recipients with valid SMS consent and valid phone availability. |
| FR-005 | System shall exclude blocked, blacklisted or communication-restricted players. |
| FR-006 | System shall deduplicate recipients within the same campaign. |
| FR-007 | System shall split recipients into configured batch sizes. |
| FR-008 | System shall publish downstream batch events with recipientRef and non-PII metadata only. |
| FR-009 | System shall log preparation results using campaignId, batchId and correlationId. |
| FR-010 | System shall prevent PII exposure in logs, events, monitoring and error messages. |

---

## 10. Business Rules

| BR ID | Rule |
|---|---|
| BR-001 | Only recipients with valid SMS consent can be included in SMS processing. |
| BR-002 | Recipients without valid phone number must be excluded. |
| BR-003 | Direct PII must not be propagated through campaign processing, broker, SlimLink integration or monitoring layers. |
| BR-004 | Recipient references must be generated only by Player Data Service. |
| BR-005 | Recipient references must be non-reversible outside authorized Player Data / Sender boundary. |
| BR-006 | The same player must not be included more than once in the same campaign. |
| BR-007 | Downstream SMS Sender resolves contact details only at the final sending step. |

---

## 11. Acceptance Criteria

### AC-001: Eligible recipients are prepared without exposing PII

**Given** an SMS campaign with selected audience,  
**When** recipient preparation starts,  
**Then** system receives privacy-safe recipient references from Player Data Service,  
**And** campaign batches do not contain phone number, email, name or other direct PII.

### AC-002: Only valid SMS recipients are included

**Given** selected audience contains players with different consent and phone states,  
**When** Player Data Service prepares recipient references,  
**Then** only players with valid SMS consent and valid phone availability are included.

### AC-003: Duplicate players are removed

**Given** the same player appears multiple times in the selected audience,  
**When** recipient batches are prepared,  
**Then** this player is included only once in the campaign recipient list.

### AC-004: Batches respect configured limits

**Given** eligible recipient count exceeds configured batch size,  
**When** preparation is completed,  
**Then** system creates multiple batches,  
**And** no batch exceeds configured size.

### AC-005: Downstream events contain only safe data

**Given** recipient batches are created,  
**When** batch events are published,  
**Then** events contain only campaignId, batchId, recipientRef and allowed non-PII metadata.

## 18. Interview Pitch — 1–2 минуты

В этой Story я сделал акцент на том, что в SMS campaign pipeline не нужно протаскивать персональные данные через все CRM-модули. Вместо передачи phone number, email или имени мы используем privacy-safe recipient reference — условный user hash / recipientRef, который генерирует Player Data Service.

Player Data Service остается владельцем персональных данных: он проверяет eligibility, phone availability, SMS consent, blacklist/suppression rules и возвращает в campaign pipeline только безопасные references. Дальше CRM работает с batch-ами, где есть campaignId, batchId и recipientRef, но нет прямых PII.

Это решение лучше, чем просто передавать PII в зашифрованном виде, потому что encrypted PII всё равно распространяет персональные данные по сервисам и требует key management. А recipientRef реализует data minimization: промежуточные сервисы не знают персональные данные, а финальный SMS Sender резолвит recipientRef только на последнем шаге перед отправкой.

Так мы одновременно решаем задачу масштабируемой batch processing, снижаем compliance-риск, уменьшаем нагрузку на pipeline и упрощаем контроль логов, мониторинга и доступа.
