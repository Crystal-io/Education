# 07. Тестирование и интеграционные взаимодействия

## Цель модуля
Сформировать у System / Business Analyst **целостное понимание тестирования** как части SDLC:
- какие существуют подходы и уровни тестирования
- как тестирование связано с требованиями и интеграциями
- какую роль играет аналитик
- как выстраивается последовательность тестирования
- как уверенно отвечать на вопросы на интервью

Фокус — **процесс и мышление аналитика**, а не ручное тестирование или автоматизация.

---

## 1. Зачем аналитику разбираться в тестировании

Тестирование — это не финальный этап, а **механизм проверки требований**.

Аналитик отвечает за:
- корректность бизнес-логики
- полноту сценариев
- согласованность ожиданий бизнеса и реализации

Если требования плохие — тестирование не спасёт продукт.

---

## 2. Основные подходы к тестированию

### 2.1 Black-box testing
- тестируется поведение системы
- без знания внутренней реализации
- основной подход для аналитика

### 2.2 White-box testing
- тестирование с учётом внутренней логики
- чаще зона разработчиков

### 2.3 Gray-box testing
- частичное знание реализации
- часто используется в интеграционных сценариях

---

## 3. Уровни тестирования (Testing Levels)

### 3.1 Unit Testing
- тест отдельных функций и компонентов
- зона ответственности разработчиков

### 3.2 Integration Testing
- проверка взаимодействия между модулями и сервисами
- критично для distributed systems

### 3.3 System Testing
- проверка системы целиком
- соответствие требованиям

### 3.4 End-to-End (E2E)
- тестирование пользовательских сценариев
- через UI или API

### 3.5 User Acceptance Testing (UAT)
- подтверждение бизнес-ценности
- финальное решение о готовности

---

## 4. Последовательность тестирования в SDLC

Типовая цепочка:

1. Unit tests (Dev)
2. Integration tests (Dev / QA)
3. System tests (QA)
4. E2E tests (QA / Product)
5. UAT (Business / Analyst)

Важно:
- этапы могут пересекаться
- аналитик вовлечён начиная с integration/system testing

---

## 5. Интеграционное тестирование: что важно учитывать

### 5.1 Типы интеграций
- sync (REST API)
- async (webhooks, queues)
- batch (файлы, отчёты)

### 5.2 Что тестировать в интеграциях
- корректность контрактов
- мэппинг данных
- обработку ошибок
- таймауты и ретраи
- идемпотентность
- повторную доставку событий

---

## 6. Инструменты тестирования (с точки зрения аналитика)

### 6.1 API и интеграции
- Postman
- Swagger / OpenAPI
- Mock-сервисы

### 6.2 Network и отладка
- Charles Proxy
- DevTools

### 6.3 Данные
- SQL
- админ-панели
- отчёты

Аналитик использует инструменты **для валидации сценариев**, а не для автоматизации.

---

## 7. Связь требований и тестирования

### 7.1 Acceptance Criteria (AC)
- основа для тестирования
- формируются аналитиком
- проверяются QA и бизнесом

### 7.2 Traceability
- требование → тест → дефект
- позволяет контролировать покрытие

Хорошая практика — поддерживать **Requirement Traceability Matrix** (RTM).

---

## 8. Типовые дефекты, которые должен видеть аналитик

- нарушение бизнес-правил
- некорректные статусы
- отсутствие обработки edge cases
- несоответствие AC
- проблемы в интеграционных сценариях

---

## 9. Best practices тестирования для System Analyst

- начинать думать о тестировании на этапе требований
- описывать позитивные и негативные сценарии
- явно фиксировать edge cases
- учитывать интеграции и асинхронность
- участвовать в UAT лично

---

## 10. Типовые вопросы на интервью (с ответами)

**Вопрос:** В чём разница между QA testing и UAT?
**Ответ:** QA проверяет техническую корректность, UAT — соответствие бизнес-ожиданиям.

**Вопрос:** Какова роль аналитика в тестировании?
**Ответ:** Формировать AC, проверять бизнес-логику, участвовать в UAT и анализе дефектов.

**Вопрос:** Что важнее: тест-кейсы или acceptance criteria?
**Ответ:** AC важнее, так как они отражают бизнес-ожидания.

**Вопрос:** Как тестировать интеграции?
**Ответ:** Через контрактные тесты, проверку ошибок, ретраев, идемпотентности и данных.

**Вопрос:** Когда фича готова к релизу?
**Ответ:** Когда выполнены AC, пройден UAT и нет критичных дефектов.
