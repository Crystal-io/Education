# 08. UML и BPMN в работе System / Business Analyst

## Цель модуля
Сформировать у System / Business Analyst **практическое и осознанное понимание UML и BPMN**:
- когда и зачем использовать каждую нотацию
- на каких этапах анализа они применяются
- какие типы диаграмм реально нужны аналитику
- как не «перерисовывать архитектуру», а усиливать мышление команды
- как уверенно отвечать на вопросы на интервью

Фокус — **инструменты анализа и коммуникации**, а не формальное моделирование.

---

## 1. Зачем аналитику UML и BPMN

UML и BPMN — это:
- инструменты **структурирования мышления**
- способ синхронизации бизнеса и разработки
- язык обсуждения сложной логики без кода

Аналитик использует диаграммы, когда:
- текст становится неоднозначным
- сценариев слишком много
- есть интеграции и переходы состояний
- нужно быстро договориться со стейкхолдерами

---

## 2. Этапы анализа и используемые нотации

| Этап | Цель | Основные инструменты |
|----|----|----------------------|
| Discovery | Понять проблему | BPMN (L0–L1), Context Diagram |
| Analysis | Разобрать логику | BPMN L2, Use Case, Activity |
| Design | Уточнить взаимодействия | UML Sequence, State Diagram |
| Delivery | Поддержка разработки | Sequence, ER, API diagrams |
| UAT | Валидация сценариев | BPMN, Use Case |

---

## 3. BPMN: когда и зачем

### 3.1 Что решает BPMN
BPMN лучше всего подходит для:
- бизнес-процессов
- пользовательских сценариев
- оркестрации действий
- описания ролей и ответственности

Если есть вопрос «**что происходит дальше?**» — это BPMN.

---

### 3.2 Уровни BPMN (L0–L3)

**L0 — Overview**
- очень высокий уровень
- для бизнеса и топ-менеджмента

**L1 — End-to-End**
- основные шаги
- роли и системы

**L2 — Detailed**
- условия
- альтернативы
- ошибки

**L3 — Technical**
- редкий уровень
- почти псевдокод

Аналитик чаще всего работает с **L1–L2**.

---

### 3.3 Ключевые элементы BPMN

- Events (Start / End)
- Tasks
- Gateways (XOR / AND)
- Pools & Lanes
- Message Flows

**Best practice:**
- один Start Event
- понятные названия действий
- минимизация «спагетти»

---

## 4. UML: когда и зачем

UML используется, когда нужно ответить на вопрос:
**«Как именно взаимодействуют системы и компоненты?»**

---

### 4.1 Основные UML-диаграммы для аналитика

#### Use Case Diagram
- показывает функциональность системы
- роли и их возможности
- используется на ранних этапах

#### Sequence Diagram
- показывает порядок вызовов
- идеально для API и интеграций

#### Activity Diagram
- похожа на BPMN
- используется реже

#### State Diagram
- жизненный цикл сущности
- статусы и переходы

---

## 5. BPMN vs UML: как выбрать

| Критерий | BPMN | UML |
|-------|------|-----|
| Процессы | Да | Ограниченно |
| Интеграции | Частично | Да |
| API | Нет | Да |
| Бизнес | Да | Слабо |
| Разработка | Средне | Отлично |

**Утверждение для интервью:**  
> BPMN — про *что* происходит, UML — про *как* это реализовано.

---

## 6. Типовые ошибки аналитиков

- рисуют BPMN там, где нужен Sequence
- слишком детализируют диаграммы
- дублируют текст документации
- не согласуют нотацию с командой

Хорошая диаграмма:
- упрощает обсуждение
- не требует пояснений

---

## 8. Связь диаграмм с требованиями

Диаграммы должны:
- поддерживать требования
- объяснять сложные места
- помогать тестированию и UAT

Best practice:
- ссылаться на диаграммы из требований
- обновлять при изменениях логики

---

## 9. Типовые вопросы на интервью (с ответами)

**Вопрос:** Когда вы используете BPMN?
**Ответ:** Когда нужно описать бизнес-процесс или пользовательский сценарий.

**Вопрос:** Когда лучше использовать UML Sequence?
**Ответ:** Для описания взаимодействий между системами и API.

**Вопрос:** Нужно ли рисовать всё?
**Ответ:** Нет, диаграммы нужны там, где текст становится неоднозначным.

**Вопрос:** BPMN или UML — что лучше?
**Ответ:** Это разные инструменты для разных задач.

**Вопрос:** Какие диаграммы вы используете чаще всего?
**Ответ:** BPMN L1–L2 и UML Sequence.
